The Solution

A fully automated B2B onboarding
pipeline with built-in AI and GDPR.

I built this in Make.com with 15 modules. Every step is explicit, logged and recoverable.

  • Detects new B2B client row in Google Sheets automatically
  • Cleans and normalizes email and company name before anything else
  • Masks all PII — AI only ever sees anonymized business signals
  • Groq AI (llama-3.3-70b-versatile) classifies client risk as Low, Medium or High
  • Routes Slack alert to the correct channel based on risk level
  • Checks for duplicate customer in Fortnox before creating
  • Creates Fortnox customer and draft invoice automatically
  • Sends personalized welcome email to the new client
  • Logs every run to Google Sheets audit log with timestamp
  • Error handler catches failures and notifies the team silently

Challenges Solved

The decisions that make
this system production-grade.

Connecting tools is straightforward. Making the system reliable under real-world conditions is where the design thinking happens.

Challenge

GDPR compliance stops AI calls

Sending real client names and emails to Groq AI would violate GDPR. I built a masking module that replaces all PII with tokens before the AI call. The AI classifies risk based on business signals only — no personal data ever leaves the system.
Challenge

Fortnox duplicate customers

Test runs created duplicate customer records in Fortnox because the system didn't check for existing entries. I added a duplicate check module that queries Fortnox for existing customers using org_number before creating a new one.
Challenge

Risk classification accuracy

The first version of the AI prompt produced inconsistent risk levels. I refined the prompt with explicit criteria (revenue, employee count, industry, country) and enforced a strict JSON output format. Accuracy improved from ~70% to ~95%.
Challenge

No audit trail for onboarding

Without logging, we couldn't answer "was this client onboarded?" I added a Google Sheets audit log that records every run: timestamp, client name (masked), risk level, Fortnox customer ID, and status. Full traceability.

What I Learned

The decisions that made
this production-ready.

Building this forced me to think beyond the happy path — every edge case had to be handled explicitly.

01

GDPR compliance changes your architecture

When I first designed the flow, I sent full client data to Groq AI. Then I redesigned — adding the masking module meant rethinking the entire data flow. Privacy-by-design is a structural decision, not a setting.

02

AI is a tool, not an authority

The risk classification improves with better prompt design. I learned that the quality of the AI's output depends almost entirely on how precisely you define the input context and the expected output format.

03

Fortnox is the hardest integration

Swedish accounting software has strict API requirements around org numbers and customer IDs. The duplicate check before creation was a lesson learned from test runs that created duplicate customers.

04

Audit logs are for debugging, not compliance

I built the audit log primarily as a compliance feature. I ended up using it constantly during development to trace exactly where a run failed — it became the most valuable debugging tool in the system.

What I'd build next: Multi-language welcome emails based on the client's country field, a retry mechanism for failed Fortnox calls, a weekly digest email with onboarding success rate by risk tier, and a Slack command that lets the sales team re-trigger onboarding for any client manually.
